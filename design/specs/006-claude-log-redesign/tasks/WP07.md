# WP07: JSONL corruption warnings and --limit

## Scope

Add two minor improvements: warn on corrupt JSONL lines (currently silently skipped), and add `--limit` to `show` and `extract` commands.

## Files
- `bin/claude-log` ŌĆö update `iter_entries()` (L226-240) to count and report skipped lines; add `--limit`/`-l` to `show` and `extract` argparse; wire limit through `cmd_show()` and `cmd_extract()`
- `tests/test_claude_log.py` ŌĆö add tests for corruption warnings and `--limit` behavior

## Steps
1. Update `iter_entries()` to track skipped lines per call: count `json.JSONDecodeError` exceptions. After iteration completes (or via a wrapper), if >1 line was skipped, emit `f"Warning: skipped {count} corrupt lines in {path}"` to stderr. A single corrupt last line (common for active sessions being written to) should NOT trigger the warning ŌĆö only warn when count > 1.
2. Implementation approach: since `iter_entries` is a generator, use a context-manager wrapper or post-iteration callback. Alternatively, move the warning to callers that exhaust the generator. Simplest: make `iter_entries` yield entries and track count internally, then print warning in `__del__` or use `itertools` wrapping. Preferred: add a thin wrapper function `iter_entries_with_warnings(path, ...)` that callers use, which exhausts the generator and prints warnings.
3. Add `--limit` / `-l` (type `_positive_int`, default `None`) to `show` argparse (after L1607)
4. Add `--limit` / `-l` (type `_positive_int`, default `None`) to `extract` argparse (after L1679)
5. Wire `--limit` in `cmd_show()`: cap the result list before printing JSON
6. Wire `--limit` in `cmd_extract()`: pass through to `_extract_tools`, `_extract_usage`, `_extract_messages` ŌĆö cap output entries
7. Add tests:
   - JSONL file with 3+ corrupt lines ŌåÆ stderr warning
   - JSONL file with 1 corrupt last line ŌåÆ no warning
   - `show --limit 5` returns at most 5 entries
   - `extract --tools --limit 10` returns at most 10 tool calls
8. Run the test suite

- `show --limit 10` returns at most 10 entries
- `extract --tools --limit 50` returns at most 50 tool calls
- All tests pass

## Avoid
- Warning on every corrupt line individually ŌĆö batch into a single count-based warning per file
- Making `--limit` default to a number ŌĆö default should be unlimited (`None`) so existing behavior is preserved
- Breaking the `iter_entries` generator contract ŌĆö callers that don't use the warning wrapper should still work
