## Objectives & Success Criteria

The `skills/mine.address-pr-issues/SKILL.md` is fully rewritten to implement the redesigned 4-phase flow:
- Phase 1 uses `git-platform` for detection and `gh-pr-threads --json` / `ado-pr-threads list --json` for thread fetching
- Phase 2 merges triage and planning, assigns investigation depth per issue, presents full plan with opt-out gate
- Phase 3 uses serial general-purpose subagents with temp file output, per-group commits, single push, then thread replies with idempotency markers
- Phase 4 provides a structured summary
- No inline GraphQL anywhere in the SKILL.md
- Thread resolution uses `gh-pr-reply --resolve` (colocation) and respects bot-vs-human policy via `__typename`
- Anti-rationalization guardrail is included in subagent prompt template
- Pre-flight warnings for draft PRs and CHANGES_REQUESTED
- Outdated thread triage uses cite-or-escalate rule
- Merge conflict strategy asks user preference (merge vs rebase)
- All open questions resolved per design doc decisions
- Estimated ~250-300 lines (down from current 377)

## Subtasks

1. Read the current `skills/mine.address-pr-issues/SKILL.md` fully
2. Read the approved design doc at `design/specs/010-address-pr-redesign/design.md` fully — this is the authoritative source for all decisions
3. Write the new SKILL.md following the structure outlined in the design doc's "SKILL.md structure (post-redesign)" section:

   **Phase 1: Fetch & Detect**
   - Platform detection via `git-platform` (check for `unknown`, stop with message)
   - PR metadata fetch: thin dispatch (GitHub: `gh pr view --json number,title,url,baseRefName,headRefName,mergeable,mergeStateStatus,statusCheckRollup,isDraft,reviewDecision`; ADO: `ado-pr show --json`)
   - Mergeable UNKNOWN retry: up to 3 times with backoff (3s, 6s, 12s), warn user if still unknown
   - Thread fetch: `gh-pr-threads {PR} --json --all` (GitHub) / `ado-pr-threads list --json` (ADO)
   - CI status: thin dispatch (GitHub: from statusCheckRollup; ADO: `az repos pr policy list`)
   - General PR comments: `gh pr view --json comments` (GitHub) / already in thread list (ADO)
   - Pre-flight warnings: isDraft, reviewDecision == CHANGES_REQUESTED

   **Phase 2: Triage & Plan**
   - Categorize issues (review comments, CI failures, merge conflicts)
   - For each review comment: check if already addressed (current code matches request)
   - Outdated thread assessment: cite-or-escalate (cite specific line or categorize as needs-manual-review; auto-categorize deleted locations)
   - General comment assessment (actionable vs discussion)
   - Assign investigation depth per issue: `light` / `medium` / `deep`
   - Present numbered plan with proposed actions and resolution policy per thread
   - Opt-out AskUserQuestion (multiSelect: "Looks good", "Skip specific items", "Cancel")
   - If conflicts: AskUserQuestion for merge vs rebase strategy

   **Phase 3: Execute**
   - Create temp dir: `get-skill-tmpdir mine-address-pr`
   - For each logical group (serial):
     - Launch general-purpose subagent with: comment(s), file paths, investigation depth, anti-rationalization guardrail, test execution mandate, temp file output path (`<tmpdir>/group-N/result.md`)
     - Main agent reads one-line summary from result file
   - After all subagents: run code-reviewer + integration-reviewer (parallel)
   - Commit per logical group with descriptive messages
   - Push once
   - After push: reply to threads with `<!-- addressed-pr-issues -->` marker, idempotency check (search for marker in thread comments), resolve bot threads via `gh-pr-reply --resolve` (GitHub) or `ado-pr-threads reply` + `ado-pr-threads resolve` (ADO), reply-only for human threads
   - 1-second delay between mutative API calls

   **Phase 4: Summary**
   - Threads resolved/replied counts
   - Code changes committed (list commits)
   - CI status (pending re-run)
   - Items skipped or needing manual review

   **Helper Scripts reference**
   - Cross-reference to `mine.gh-tools` skill and `ado-pr-threads --help`

4. Verify the SKILL.md frontmatter matches the current format (name, description, user-invocable)
5. Verify the "When to Activate" trigger phrases match the routing table in `rules/common/capabilities.md`
