## Objectives & Success Criteria

`rules/common/findings.md` is rewritten to describe the new Resolution Manifest flow end-to-end. The rule remains mechanism-agnostic (zero tmux references, zero editor-specific invocations) and delegates all editor-launching mechanics to `bin/edit-manifest` by name. The rewrite grows the file from ~54 lines to ~150 lines.

**Required content sections, in order**:

1. **Principle: All Findings Must Be Resolved** — kept verbatim from current version
2. **Presenting Findings** — kept verbatim from current version
3. **Verb Vocabulary** — new canonical definition of the six-verb vocabulary (`fix`, `file`, `defer`, `skip`, `ask`, `A`/`B`/`C`) as a single authoritative table. Other files (mine.challenge Phase 4, anti-pattern catalog entries) reference this by name rather than re-stating verbs.
4. **Default Verb Selection** — recommendation-based rule using the contract field `recommendation:`. Pre-populates the manifest from finding resolution class and recommendation content. Includes a Format-version 1 fallback rule ("if `recommendation:` is absent, default to `ask`").
5. **Resolution Manifest** — describes the markdown file format (per-finding blocks with `**Verb:**` lines, header with usage instructions and pre-hash comment). Includes the example manifest block from the design doc.
6. **Consent Gate** — lightweight "ready to review the resolution manifest?" AskUserQuestion before the editor launches. This is the rename/repositioning of the old Proceed Gate; preserves the term "Proceed Gate" as an alias in the prose for backward compatibility with callers that still reference it.
7. **Editor Session** — **mechanism-agnostic** description: "the skill calls `edit-manifest <path>`, waits for the session to complete (via task-notification or synchronous return), and proceeds." NO tmux references, NO nvim autocmds, NO shell-quoting details. Those all live in `bin/edit-manifest` (per WP01).
8. **Detection Logic** — four-way decision tree based on hash comparisons (original hash, shadow hash, exit behavior). Includes the full decision table from the design doc.
9. **Commit Gate** — single binary AskUserQuestion: "Execute resolution manifest?" with Yes / Revise / No options.
10. **Execution** — two-phase iteration:
    - Phase 1: immediate verbs (`fix`, option letters, `ask` with `(N/M)` position counter, `defer`, `skip`)
    - Phase 2: batched `file` verbs at the end via `gh-issue create`
    - Phase 3: summary and return
11. **Manifest Validation Spec** — 7 concrete validation rules covering unknown verbs, modified IDs, option-letter mismatches, deleted rows, added rows, row count anomalies, and error routing (always back to Revise, never abort).
12. **Re-edit Loop Cap** — 5 iterations max, fallback to inline-display mode on exceeding
13. **Named Anti-Pattern Catalog (CRITICAL)** — modeled structurally on `rules/common/interaction.md`'s "AskUserQuestion Blocks in Skills (CRITICAL)" section. Eight enumerated failure modes, each with:
    - A concise rule statement
    - A verbatim example drawn from the session logs (`99e9716a`, `0421be8c`, `cb1c59ad`) showing what the failure looks like
    - A "do this instead" counter-example using the manifest flow
14. **Skill-Specific Overrides** — kept (mostly) unchanged. Existing note that visual-qa and tool-gaps use their own gates. Add a follow-up note: "A subsequent design will migrate visual-qa to produce manifest-compatible findings and use `edit-manifest` directly via the Resolution Manifest flow."

**Observable success**:
1. The file is rewritten and contains all 14 sections above in the specified order
2. The word "tmux" does not appear anywhere in the file (`grep -c tmux rules/common/findings.md` returns 0)
3. The verb vocabulary is defined exactly once (grep for the six verbs should find them as a table, not scattered through the file)
4. The phrase "Proceed Gate" appears as an explicit alias for the Consent Gate (so existing references in `mine.challenge/SKILL.md:32,123,126` and `mine.tool-gaps/SKILL.md:178` remain semantically valid)
5. The 8 anti-patterns are each enumerated with a concrete example — not abstract statements
6. The file references `bin/edit-manifest` by name at least once but does not describe its internal mechanism
7. Final file length is roughly 120-180 lines (sanity check — avoids accidental truncation or bloat)

## Review Guidance

**Critical correctness checks** (any failure is a blocker):

- **Mechanism-agnosticism**: The file MUST NOT contain "tmux", "popup", "new-window", "wait-for", "nvim", "display-popup", or any other editor-specific or terminal-specific term. Finding 3 from the challenge (tmux knowledge leaking into shared rule) is the structural concern this file's rewrite exists to fix. Any mechanism-specific text is a regression.
- **Canonical verb vocabulary**: The six verbs must be defined in exactly one place (the Verb Vocabulary section). Other sections MUST reference by name, not re-state. Cross-check that the execution section doesn't re-define what each verb does.
- **Proceed Gate alias preserved**: Existing callers (`mine.challenge/SKILL.md:32,123,126`, `mine.tool-gaps/SKILL.md:178`) reference "Proceed Gate" verbatim. The alias must be explicit in the rewritten text ("the Consent Gate — formerly the Proceed Gate — ...") to keep those references semantically valid. This was Finding F9's recommendation.
- **Recommendation-based defaults**: The Default Verb Selection section must use the contract field `recommendation:` — NOT `confidence:`. Finding F5 explicitly identified `confidence:` as presentation-only. Any use of `confidence:` for default selection is a regression.
- **Anti-pattern catalog is concrete**: Each of the 8 anti-patterns must include a VERBATIM example from real session logs. Abstract statements ("don't bundle findings") are insufficient — the design explicitly rejects prose-only rules as the primary defense. The catalog is defense-in-depth and must be as sticky as possible, which requires concrete examples.
- **No silent abandonment escape**: The "All Findings Must Be Resolved" principle must be preserved. The catalog must explicitly name the "Skip revisions / Enough — approve as-is" anti-pattern from the compiled examples (Anti-pattern #8). Any rewrite that weakens this principle is a regression.

**Design constraint checks** (any deviation is a warning):

- **Validation spec order matches design**: the 7 validation rules are numbered and ordered deliberately in the design — the WP doesn't need to preserve exact numbering if the reviewer prefers reordering, but all 7 rules must be present.
- **Execution-phase `(N/M)` counter** must be explicitly stated for `ask` verbs — this is Pattern 1 from prior art and the fallback mitigation for residual bundling risk (Finding F12).
- **Batched file verbs at end** must be explicit — Finding F7's resolution. The rule must clearly state that `file` verbs execute AFTER all other verbs, not inline.
