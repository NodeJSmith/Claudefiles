## Test Strategy

**N/A — no unit test infrastructure in this repo (prose-only caller updates).**

**Session-level validation** — run after WP01, WP02, and this WP's edits are complete. These validate the end-to-end integration of all three WPs:

1. **Happy-path manifest flow on a real target**: Run `/mine.challenge <some PR branch with diverse findings>` standalone. Verify the full flow:
   - Findings rendered in Phase 4
   - Consent Gate appears with "Ready to review the resolution manifest?"
   - After Yes, a manifest file is written to the challenge tmpdir
   - `edit-manifest` is invoked (tmux window opens with nvim and shadow autosave active)
   - After save/exit, detection logic runs and identifies the edit state
   - Commit Gate appears
   - After Yes, execution phase emits per-finding AskUserQuestion only for `ask` verbs (with `(N/M)` position counter)
   - `file` verbs are batched at the end
   - Summary is presented

2. **Bundling regression test**: Re-challenge the target from session `99e9716a` (hassette source-tier unification PR) which previously produced "MEDIUM batch 1 / 2 / 3" bundled prompts. The execution phase should now produce N individual AskUserQuestion calls where the user chose `ask`, NOT N bundled into 1. This is the primary regression check for the original bug.

3. **Shadow-file `:q!` recovery test**: Open the manifest via the flow, type changes, `:q!` to discard. Verify the detection logic identifies the unsaved-but-typed state and offers recovery options (recover from shadow, abandon, re-open).

4. **Non-blocking editor warning test**: Set `EDITOR=code` (without `--wait`), trigger the flow. Verify the warning appears before the tmux window opens. If the user proceeds anyway and the editor returns in <1s with unchanged hash, verify the reactive check catches it.

5. **Tertiary fallback test**: Run the skill outside tmux (fresh terminal session without tmux). Verify the fallback message prints the manifest path and instructions, and the user can edit the file manually and return. Free-text parsing of the return message resolves the manifest.

6. **Structured caller bypass test**: Run `/mine.design` on a feature directory. Verify that mine.design's own Apply-all/Cherry-pick/Skip revision flow is used — NOT the Resolution Manifest flow. No `resolutions.md` file should be written. No `edit-manifest` invocation should occur. mine.design's own flow is unchanged.

7. **TENSION handling test**: Craft a findings file with a TENSION finding, run the flow. Verify the default verb is `defer`, the manifest presents the disagreement (side-a/side-b/deciding-factor), and execution records the deferral in the session summary rather than attempting to apply a fix.

8. **Visual-qa unaffected test**: Run `/mine.visual-qa` on a UI target. Verify the existing gate (Fix issues now / Read report / Re-run / File as issues) still appears and "Fix issues now" still routes to the legacy Skill-Specific Override path — NOT the Resolution Manifest flow. The follow-up note is present but the behavior is unchanged.

9. **Anti-pattern catalog regression test**: Trigger a scenario that previously produced a bundled AskUserQuestion (e.g., 9 findings all requiring user decision). Verify the LLM reads the new `rules/common/findings.md` and follows the manifest flow instead of the old bundled pattern. This tests whether the catalog + structural mechanism together prevent the regression.

10. **`(N/M)` counter test**: Create a manifest with exactly 5 `ask` rows and 0 other user-decision rows. Confirm execution emits exactly 5 separate AskUserQuestion calls, each with `(N/5)` in the header (`(1/5)`, `(2/5)`, ..., `(5/5)`).
