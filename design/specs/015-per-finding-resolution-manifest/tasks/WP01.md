## Objectives & Success Criteria

`bin/edit-manifest <manifest-path>` is an executable shell script that opens the specified manifest file for user editing via `tmux new-window + wait-for`, with full support for:

- **Tmux detection**: detects whether `$TMUX` is set and tmux ≥ 2.4 is available; falls through to tertiary fallback when absent
- **Blocking-editor detection**: proactively warns if `$EDITOR` matches a known-non-blocking list (`code`, `subl`, `atom`, `gedit`, `open`, `xdg-open`) before launch, and reactively detects post-exit via elapsed time when the session completes in <1 second
- **Shadow-file autosave**: launches nvim with `autocmd CursorHold,CursorHoldI <buffer> silent write! <manifest>.shadow` and `autocmd VimLeave * silent write! <manifest>.shadow` so the shadow file captures in-memory buffer state regardless of how nvim exits (including `:q!`)
- **Trap-based signaling**: the child shell uses `trap "tmux wait-for -S edit-done" EXIT HUP TERM` so the parent's `wait-for` is always signaled even if the tmux window is externally closed
- **wait-for lock mode**: parent acquires `tmux wait-for -L edit-done` before `new-window` to eliminate the signal-before-listener race
- **`zsh -l -c` PATH wrapping**: the editor invocation runs under a login shell so mise/asdf/direnv/nix-managed editors resolve correctly
- **Tertiary fallback**: when tmux is absent, the script prints the manifest path and instructions, returns success, and the caller handles the free-text edit path
- **Diagnostic logging**: writes structured events to `<tmpdir>/editor-log.md` where `<tmpdir>` is the parent directory of the manifest file. Events include invocation metadata, blocking-editor check result, exit code, elapsed ms, and any errors
- **Exit codes**: 0 on successful edit session completion (regardless of save/discard), 1 on unrecoverable error

**Observable success**:
1. `edit-manifest /tmp/test-manifest.md` opens nvim in a new tmux window with shadow autosave active
2. After the user saves and exits, the script returns exit 0 and `<manifest>.shadow` exists
3. After the user exits with `:q!`, the shadow file still contains the in-memory edits
4. `edit-manifest /tmp/test.md` with `unset TMUX` prints the fallback message and returns 0
5. `edit-manifest /tmp/test.md` with `EDITOR=code` emits the known-non-blocking warning before launch
6. `<tmpdir>/editor-log.md` contains the invocation, exit, and elapsed-ms entries
7. `bin/lint-cli-conventions bin/edit-manifest` exits 0 with no reported issues

## Subtasks

1. Create `bin/edit-manifest` with bash shebang `#!/usr/bin/env bash`, `set -euo pipefail`, and a header comment block describing purpose and usage (follow the `bin/claude-tmux` pattern for structure).

2. Implement `--help` / `-h` handling that prints usage and exits 0. Usage: `edit-manifest <manifest-path>`.

3. Implement argument parsing: require exactly one positional argument (manifest path). If missing or file doesn't exist, print error to stderr and exit 1.

4. Derive `SHADOW="${MANIFEST_PATH}.shadow"` and `LOG="$(dirname "$MANIFEST_PATH")/editor-log.md"`. Delete any stale shadow file at `$SHADOW`.

5. Define a logging helper function `log_event()` that appends a structured entry to `$LOG`. Each entry is a timestamp + event type + key details (e.g., `2026-04-09T13:45:00Z invocation manifest=/tmp/foo.md editor=nvim tmux=yes`).

6. Implement tmux availability detection: if `[ -z "${TMUX:-}" ]`, log `tmux_absent` and jump to tertiary fallback. Otherwise check `tmux -V` parses as version ≥ 2.4 (wait-for is old, widely available).

7. Implement known-non-blocking `$EDITOR` check: match the first word of `$EDITOR` (after resolving `${EDITOR:-nvim}`) against the list `code subl atom gedit open xdg-open`. If matched, print a warning to stderr with a suggested fix (`EDITOR='code --wait'` etc.) and log `blocking_editor_warning editor=<value>`. Do not abort — user may have already configured `--wait` flags.

8. Implement the primary tmux path:
   - Compute pre-hash: `PRE_HASH=$(sha256sum "$MANIFEST_PATH" | cut -d' ' -f1)`
   - Acquire the wait-for lock in background: `tmux wait-for -L edit-done &` and capture `LOCK_PID=$!`
   - Launch the edit window:
     ```bash
     tmux new-window -n manifest "zsh -l -c '
       trap \"tmux wait-for -S edit-done\" EXIT HUP TERM
       nvim \
         -c \"set updatetime=2000\" \
         -c \"autocmd CursorHold,CursorHoldI <buffer> silent write! ${SHADOW}\" \
         -c \"autocmd VimLeave * silent write! ${SHADOW}\" \
         \"${MANIFEST_PATH}\"
     '"
     ```
   - Wait for the child signal: `wait "$LOCK_PID"` (the lock releases when -S fires)
   - Release the lock: `tmux wait-for -U edit-done 2>/dev/null || true`
   - Capture elapsed time with a `SECONDS` or `date +%s%N` bracket around the launch+wait.

9. Implement tertiary fallback path:
   - Print to stdout: `"No tmux detected. The manifest is at <path>. Open it in your editor, edit the **Verb:** lines, save, and return here to continue."`
   - Log `tertiary_fallback_used`
   - Exit 0

10. Implement post-edit elapsed-time check for non-blocking editor detection: if elapsed ms < 1000 AND (hash unchanged OR blocking warning was emitted), log `non_blocking_editor_detected elapsed=<ms>`. The caller (findings.md Phase 4 detection logic) will handle the user-facing re-prompt; the script just logs for diagnostics.

11. Add `chmod +x bin/edit-manifest` (make the script executable).

12. Verify `install.sh` picks up the new script automatically. `install.sh` lines 97-112 glob `bin/*` and symlink each file into `~/.local/bin/` — the new script should auto-install on next `./install.sh` run. Run `./install.sh` locally and verify `ls -l ~/.local/bin/edit-manifest` shows a symlink.

13. Update `README.md` inventory if it contains a list of `bin/` scripts (check for a section listing CLI tools; add `edit-manifest` if found). This is the "always update README.md when adding bin/ scripts" rule from CLAUDE.md.

14. Run `bin/lint-cli-conventions bin/edit-manifest` if the linter applies to shell scripts; fix any reported issues.

## Test Strategy

**N/A — no unit test infrastructure in this repo (prompt/rules/bin repo).**

**Manual validation scenarios** (run after the script is written):

1. **Happy path in tmux**: Create a throwaway test manifest at `/tmp/test-manifest.md` with a few `**Verb:**` lines. Run `edit-manifest /tmp/test-manifest.md` from inside a tmux session. Verify: (a) new tmux window opens with nvim loaded, (b) the file is editable, (c) `:wq` exits cleanly, (d) `/tmp/test-manifest.md.shadow` exists and matches the saved file, (e) the parent bash call returns exit 0, (f) `/tmp/editor-log.md` contains the invocation and exit entries.

2. **`:q!` recovery**: Open the same file via the script, type a few changes, then `:q!` to discard. Verify: (a) the original file is unchanged, (b) the shadow file contains the typed changes, (c) the parent bash call still returns exit 0.

3. **Tmux window closed externally**: Open via the script, then from another pane run `tmux kill-window -t manifest`. Verify: (a) the trap fires and signals `edit-done`, (b) the parent unblocks and returns exit 0 within ~1 second.

4. **No tmux fallback**: Run `unset TMUX && edit-manifest /tmp/test.md` in a plain shell. Verify: (a) the script prints the fallback message with the file path, (b) exits 0 immediately, (c) no nvim window opens, (d) log entry shows `tertiary_fallback_used`.

5. **Non-blocking editor warning**: Run `EDITOR=code edit-manifest /tmp/test.md` inside tmux. Verify: (a) warning is printed to stderr suggesting `code --wait`, (b) the script still proceeds to launch (user may have configured `code --wait` alias elsewhere), (c) log entry shows `blocking_editor_warning editor=code`.

6. **PATH resolution via login shell**: Confirm the inner `zsh -l -c` wrapper resolves `nvim` from mise's install path. This was verified in the dogfood session — replicate in a fresh terminal without `claude` running to confirm.

7. **Missing argument**: `edit-manifest` (no args) should exit 1 with a usage message. `edit-manifest /nonexistent/file.md` should exit 1 with "file not found" error.

8. **Help flag**: `edit-manifest --help` and `edit-manifest -h` should print usage and exit 0.

None of these require external test infrastructure — all are manual shell invocations producing observable file-system and exit-code signals. Each scenario reproduces a specific behavior validated during the design dogfood (documented in design.md § Test Strategy).

## Review Guidance

**Critical correctness checks** (any failure is a blocker):

- **Shell quoting correctness**: The nested `zsh -l -c` + nvim `-c` wrapping is the highest-risk area. Verify that the nvim command lines `autocmd CursorHold,CursorHoldI <buffer> silent write! ${SHADOW}` survive all layers of quoting with spaces intact. The dogfood session observed that naive invocations fail with "command not found" or split arguments wrong.
- **`zsh -l -c` is used**, not bare `zsh -c` or `bash -lc` — the login-shell semantics for PATH resolution were verified in dogfood; deviating risks regressions for mise/asdf/direnv users.
- **Trap signals EXIT, HUP, and TERM** — not just EXIT. The design explicitly requires this for the "user closed tmux window externally" case (finding N3).
- **Wait-for lock is acquired BEFORE `new-window`** — the lock-then-signal ordering is load-bearing for race safety (finding N2). Reviewing `tmux wait-for -L` as a backgrounded process before `new-window` is the correct pattern.
- **Shadow file path is derived from the manifest path**, not hard-coded — `${MANIFEST_PATH}.shadow` sibling to the manifest.
- **Editor-log.md is in the manifest's parent directory**, not a hard-coded path — `$(dirname "$MANIFEST_PATH")/editor-log.md`. This ensures logs live alongside the manifest in the caller's tmpdir.
- **No `eval` usage** — the CLAUDE.md convention and the bash-safety rule prohibit eval. If dynamic command construction is needed, use arrays with `"${args[@]}"` expansion, not eval.

**Design constraint checks** (any deviation is a warning):

- **The script is mechanism-specific** (tmux, nvim autocmds). This is INTENDED per design — the whole point of the helper script is to encapsulate tmux specifics so they don't leak into `rules/common/findings.md`. Keeping tmux knowledge here is correct.
- **The script is callable as a plain CLI** — no Claude-specific wrapping, no skill assumptions. Future skills beyond mine.challenge should be able to invoke it.
- **The tertiary fallback does NOT attempt to launch any editor** — it just prints instructions and exits. The caller handles the "user edits and returns" loop via chat.
- **Exit codes are meaningful**: 0 on "edit session completed" (even if user didn't change anything), 1 on invocation error, 1 on unrecoverable script error. The design's detection logic relies on distinguishing these.

**Suggest reviewer run the 8 manual validation scenarios listed under Test Strategy** and verify each produces the documented behavior.
