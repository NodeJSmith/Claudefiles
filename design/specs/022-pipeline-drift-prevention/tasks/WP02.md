## Objectives & Success Criteria

- The `wp-move`, `wp-list`, `status`, and `design-extract` commands are removed from spec-helper
- The `validate` command accepts `T*.md` files with the new frontmatter schema (`task_id`, `title`, `depends_on`, `implements`)
- The `archive` command globs `T*.md` instead of `WP*.md` and also finds `context.md` for deletion
- The `checkpoint` commands accept task IDs in `T01`/`T02` format alongside the existing `WP01`/`WP02` format (backward-compatible during transition)
- All existing tests for removed commands are deleted; new tests cover the updated schema validation
- `spec-helper --help` shows only the retained commands: `init`, `validate` (renamed from `wp-validate`), `archive`, `checkpoint-init`, `checkpoint-read`, `checkpoint-update`, `checkpoint-verdict`, `checkpoint-delete`, `next-number`

## Subtasks

1. In `packages/spec-helper/src/spec_helper/validation.py`: update `WP_ID_PATTERN` to match both `WP\d{2,}` and `T\d{2,}` formats (transition period). Add `TASK_ID_PATTERN = re.compile(r"^T\d{2,}$")` for the new canonical format. Update `CANONICAL_FIELDS` to `{"task_id", "title", "depends_on", "implements"}`. Keep `OLD_SCHEMA_FIELDS` and add `"work_package_id"` and `"lane"` to it (these are now old-schema). Add a `normalize_task_metadata()` function that maps `work_package_id` → `task_id` and drops `lane`. Update `validate_wp_metadata()` to also accept `task_id` field. Rename to `validate_task_metadata()`.
2. In `packages/spec-helper/src/spec_helper/filesystem.py`: rename `read_wp_files()` → `read_task_files()` with the glob changed from `WP*.md` to match both `WP*.md` and `T*.md` (transition). Rename `find_wp_file()` → `find_task_file()` with similar dual-pattern support. Update `normalize_wp_metadata` import to `normalize_task_metadata`.
3. In `packages/spec-helper/src/spec_helper/checkpoint.py`: update `_validate_verdict()` to accept both `WP\d+` and `T\d+` ID patterns. Update `_parse_checkpoint()` verdict block parsing to accept both `WP\d+` and `T\d+` headers.
4. In `packages/spec-helper/src/spec_helper/cli.py`: remove the `wp-move`, `wp-list`, `status`, and `design-extract` subcommand parsers and their dispatch entries. Rename `wp-validate` to `validate` (keep `wp-validate` as a hidden alias for backward compatibility during transition). Update all imports to use new function names.
5. In `packages/spec-helper/src/spec_helper/commands.py`: remove `cmd_wp_move`, `cmd_wp_list`, `cmd_status`, `cmd_design_extract` functions and their helpers (`_REVIEWER_SECTIONS`, `_PRIMARY_SECTION`). Rename `cmd_wp_validate` → `cmd_validate`. Update the validation function to use `normalize_task_metadata` and `validate_task_metadata`. Update `cmd_archive` to use `read_task_files()` and handle both `WP*.md` and `T*.md` globs. Update `cmd_checkpoint_verdict` to normalize both `WP` and `T` prefixed IDs.
6. In `packages/spec-helper/tests/test_cli.py`: remove tests for `wp-move`, `wp-list`, `status`, and `design-extract` commands. Add tests for the `validate` command with new `T*.md` files using `task_id`/`implements` frontmatter. Add tests verifying backward compatibility with `WP*.md` files.
7. In `packages/spec-helper/tests/test_checkpoint.py`: add test cases for `T01`-format task IDs in verdict blocks. Verify parsing handles both `WP01` and `T01` headers.
8. In `packages/spec-helper/tests/test_archive.py`: update test fixtures to include `T*.md` files alongside or replacing `WP*.md` files. Verify archive globs find both formats.

## Test Strategy

Tests in `packages/spec-helper/tests/`:

- `test_cli.py::test_validate_new_task_schema` — validate a `T01-foo.md` file with `task_id: T01`, `implements: ["FR#1"]` frontmatter; expect PASS
- `test_cli.py::test_validate_old_wp_compat` — validate a `WP01.md` file with old `work_package_id` frontmatter; expect PASS with normalization warnings
- `test_cli.py::test_validate_missing_task_id` — validate a `T01.md` with no `task_id`; expect auto-derived from filename
- `test_cli.py::test_validate_implements_format` — validate `implements: ["FR#1", "AC#2"]`; expect PASS. Validate `implements: ["FR-1"]`; expect error
- `test_cli.py::test_removed_commands_absent` — verify `wp-move`, `wp-list`, `status`, `design-extract` are not in the parser
- `test_checkpoint.py::test_verdict_t_format_id` — append a verdict with `T01` ID; verify it round-trips through read
- `test_archive.py::test_archive_task_files` — create `T01.md`, `T02.md` with `lane: done` equivalent, verify archive removes `tasks/` directory

Run with: `timeout 300 uv run --directory packages/spec-helper pytest -v`

## Review Guidance

- **Spec reviewer**: Verify that all four commands (`wp-move`, `wp-list`, `status`, `design-extract`) are fully removed — no dead code, no dangling imports. The `validate` command must accept both old and new schemas during transition.
- **Code reviewer**: Check that `normalize_task_metadata()` creates new dicts (immutability rule). Verify regex patterns are anchored (`^...$`). The `implements` field validation should check canonical `FR#N` / `AC#N` format per FR#13a.
- **Integration reviewer**: Verify no other file in the repo imports the removed functions. Check that `archive` still works with existing `WP*.md` specs that haven't been re-planned yet (backward compat). Verify the `wp-validate` → `validate` alias is registered.
- **FAIL if**: Any removed command is still importable. The `implements` field accepts non-canonical identifiers. Tests don't cover both old and new ID formats.
