## Objectives & Success Criteria

- The design doc template in `skills/mine.define/SKILL.md` produces Functional Requirements with `FR#N` identifiers and Acceptance Criteria with `AC#N` identifiers
- The template includes an optional "Visual Artifacts" section for mockup/screenshot paths
- The template includes a "Key Constraints" section for explicit anti-patterns and prohibitions
- Functional requirements are written to be individually addressable — each describes one testable behavior
- The quality validation checklist (Phase 5) verifies the new sections
- Features without visual artifacts omit the Visual Artifacts section entirely (not an empty section)

## Subtasks

1. Edit the design doc markdown template in `skills/mine.define/SKILL.md` Phase 4 ("Write design.md") to add `FR#N` numbering to each item in the `## Functional Requirements` section and `AC#N` numbering to each item in the `## Acceptance Criteria` section. Add a rule that identifiers must use canonical format (`FR#N` and `AC#N` where N is a positive integer) and be unique within the document.
2. Add a `## Visual Artifacts` section to the template between Acceptance Criteria and Dependencies. Include a rule that this section is optional — only included when visual references (mockups, screenshots, prototypes) exist. Each entry lists a file path and a description of what the artifact shows.
3. Add a `## Key Constraints` section to the template after Visual Artifacts (or after Acceptance Criteria if no Visual Artifacts). Each entry describes an explicit anti-pattern or prohibited approach specific to this feature. Include a rule that this section contains feature-specific prohibitions sourced from discovery, not general coding best practices.
4. Update the Phase 5 quality validation checklist to add items verifying: (a) all FRs have unique `FR#N` identifiers, (b) all ACs have unique `AC#N` identifiers, (c) each FR describes one testable behavior (not a compound list), (d) Key Constraints section is present (may be empty if no feature-specific prohibitions), (e) Visual Artifacts section is present only when visual references exist.
5. Update the "Rules for content" block in Phase 4 to note that FR identifiers and AC identifiers use canonical format and must be unique.

## Review Guidance

- **Spec reviewer**: Verify that the template changes match FR#1-5 and AC#1 from the design doc. The identifiers must use the exact format `FR#N` / `AC#N` (not `FR-N`, `FR.N`, or other variants). The Visual Artifacts section must be clearly marked as optional with guidance on when to include/omit it.
- **Code reviewer**: Check that the template additions are consistent with the existing template style (indentation, heading levels, rule formatting). No new markdown heading levels should be introduced.
- **Integration reviewer**: Verify the quality validation checklist items (Phase 5) are numbered consistently with existing items. Check that the new sections don't break the template's section ordering contract that downstream skills (mine.plan, mine.orchestrate) depend on.
- **FAIL if**: FR/AC identifier format doesn't match `FR#N` / `AC#N` regex. Visual Artifacts section is required instead of optional. Key Constraints section is missing from the template.
