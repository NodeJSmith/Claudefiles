## Objectives & Success Criteria

- `skills/mine.orchestrate/SKILL.md` dispatches executors using the new task file format (T*.md with Prompt/Focus/Verify sections) instead of WP files
- Executors receive: their task prompt file, the task-specific Focus section, the master context file path, and explicit design doc section references — not design-extract output
- The `design-extract` step in Phase 0 is removed (executors read design doc sections directly via references in the task file)
- Executors emit structured verification reports with DONE/CONTESTED per criterion from the task's Verify section
- CONTESTED criteria are escalated to the user before spec review runs (FR#23a)
- `skills/mine.orchestrate/spec-reviewer-prompt.md` uses binary IMPLEMENTED/NOT_IMPLEMENTED per criterion with no severity ratings (FR#25)
- `skills/mine.orchestrate/implementer-prompt.md` instructs executors to read referenced design doc sections and visual artifacts as primary sources, using the task prompt as a checklist
- The spec reviewer verifies FR/AC identifier correspondence — invented or substituted identifiers are flagged as FAIL (FR#27a)
- All WP-specific terminology and file patterns are updated to task-based equivalents throughout the SKILL.md

## Subtasks

1. In `skills/mine.orchestrate/SKILL.md` Phase 0: remove the "Generate design extracts" step (lines 110-126) that runs `spec-helper design-extract`. Executors will read design doc sections directly via references in their task files. Update "Read all WP files" to "Read all task files" — change the glob from `WP*.md` to `T*.md`, extract `task_id` instead of `work_package_id`, and remove `lane` extraction (lane tracking eliminated). Remove the dev server check's WP-specific language. Update checkpoint-init to store task IDs.
2. In Phase 1: update WP list display to show task IDs and titles (no lane column — lanes are eliminated). Auto-select start point from checkpoint state only (no lane-based selection). Update all `WP` references to `T` format.
3. In Phase 2 Step 1 ("Announce"): replace `spec-helper wp-move` calls with `spec-helper checkpoint-update` — lane transitions are replaced by checkpoint state only. Update WP ID references to task IDs throughout.
4. In Phase 2 Step 4 ("Launch executor"): rewrite the executor prompt template. Replace `## Design doc (architecture reference)` (which used design-extract output) with `## Design doc path` containing the path to design.md and a note that the executor should read referenced sections directly. Include `## Master context path` pointing to `tasks/context.md`. Include the task file's Focus section as `## Task focus`. Include the task file's Verify section as `## Verification criteria`. Keep the TDD reference, test command, and visual verification status sections.
5. In Phase 2 between Steps 4 and 5: add a new "CONTESTED criteria resolution" step. After the executor completes, parse its verification report for any CONTESTED criteria. For each CONTESTED criterion: present the executor's rationale to the user via AskUserQuestion with options "Authorize deviation (update criterion)" / "Reject (executor retries with original criterion)". If authorized, update the criterion in the task file's Verify section. If rejected, re-run the executor with the original criterion as focus. Only after all CONTESTED criteria are resolved does spec review proceed.
6. In Phase 2 Step 5 ("Parallel review pass"): update the spec reviewer prompt template. Replace the design-extract reference with the task file's Verify section (the binary checklist). Pass the full list of FR/AC identifiers from the task's `implements` field. Remove changed-files reference from WP format.
7. In Phase 2 Steps 8/8.5 ("Gate"): replace `spec-helper wp-move` calls with checkpoint updates. Remove lane transitions (`doing` → `done`, `doing` → `for_review`). The checkpoint verdict is the sole state record.
8. In Phase 3 ("Post-Execution Review"): update `spec-helper status` call in Step 1 to use checkpoint-read data directly (status command removed in WP02). Update implementation-review invocation to reference task files.
9. Rewrite `skills/mine.orchestrate/implementer-prompt.md`: replace "Reading a WP Spec" table with "Reading a Task File" table covering the new sections (Summary, Prompt, Focus, Verify). Add instructions to read design doc sections referenced in the task's Prompt section. Add instructions to read visual artifact files referenced in the Focus section. Add Verify section handling: after completing each subtask, check corresponding Verify criteria and mark DONE. If a criterion cannot be met, mark CONTESTED with rationale explaining why. Update the output format to include a `**Verification report:**` section with DONE/CONTESTED per criterion from the Verify section.
10. Rewrite `skills/mine.orchestrate/spec-reviewer-prompt.md`: replace severity-based model with binary IMPLEMENTED/NOT_IMPLEMENTED per criterion. The reviewer receives the task's Verify section (with FR/AC references) and the executor's verification report. For each criterion: independently verify by reading the implementation code, mark IMPLEMENTED or NOT_IMPLEMENTED. No WARN state for completeness — only PASS (all implemented) or FAIL (any gap). Add a check for dropped criteria (criterion in task file but absent from executor report). Add FR/AC identifier verification — any identifier in the executor's report that doesn't correspond to a criterion in the task's Verify section is flagged as FAIL. Remove all severity language (LOW/MEDIUM/HIGH/CRITICAL).
11. Update all remaining WP-specific terminology throughout the SKILL.md: "Work Package" → "task", "WP" → "task" (in prose), "WP01" → "T01" (in examples), `WP*.md` → `T*.md` (in globs), `<wp_id>` → `<task_id>` (in variable names), `wp_id` → `task_id` (in checkpoint commands).

## Review Guidance

- **Spec reviewer**: Verify the executor prompt template no longer includes design-extract output — it must reference the design doc path for the executor to read directly. Verify the CONTESTED deviation flow matches FR#23a exactly: user presented with deviation before spec review, authorized deviations update the task file. Verify the spec reviewer prompt has zero severity language — only IMPLEMENTED/NOT_IMPLEMENTED.
- **Code reviewer**: Check that the CONTESTED resolution step doesn't create an infinite loop (executor retries should have a bound — design doc says orchestrator presents to user, not auto-retries). Verify the implementer-prompt.md output format includes the verification report with DONE/CONTESTED per criterion.
- **Integration reviewer**: Verify all `spec-helper wp-move` calls are replaced with `spec-helper checkpoint-update` — no wp-move calls should remain. Verify the Phase 3 Step 1 summary no longer calls `spec-helper status`. Verify the executor prompt references `context.md` and the design doc path, not extracted text. Cross-check that the spec-reviewer-prompt.md's expected input format matches what the orchestrate SKILL.md actually passes to it.
- **FAIL if**: Any `spec-helper wp-move` or `design-extract` call remains. Spec reviewer has severity ratings. CONTESTED criteria bypass user approval. Executor doesn't receive design doc path reference.
