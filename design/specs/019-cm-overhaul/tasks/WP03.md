## Objectives & Success Criteria

Implement the v5 and v6 PRAGMA user_version migrations in `db.py`. After this WP:

- v5 recomputes `aggregated_content` (SET semantics) with file paths and commits for all existing branches, renames INTERRUPTED→ABANDONED in stored summaries, gates `_migrate_project_paths()` as a pre-pass, and runs a single FTS rebuild
- v6 truncates oversized `context_summary_json` entries (>50KB)
- Both migrations are idempotent and safe under concurrent execution
- `_migrate_project_paths()` no longer runs on every `get_db_connection()` call
- All migrations complete in under 30 seconds on a ~4000-branch database

## Subtasks

1. Extract `_migrate_project_paths()` from the unconditional code path in `get_db_connection()` — it will be called only inside the v5 migration block.

2. Implement v5 migration in `_migrate_columns()` (or a new `_run_versioned_migrations()` function if separating DDL from DML): for each branch in batches of 50, read `aggregated_content`, `files_modified`, and `commits` from the branches table; recompute aggregated_content as message text + file paths + commit text (matching the SET logic from WP02's session_ops); also string-replace `**Status:** INTERRUPTED` with `**Status:** ABANDONED` in `context_summary` and update the `disposition` field in `context_summary_json`. After all batches, run `INSERT INTO branches_fts(branches_fts) VALUES('rebuild')`. Bump `PRAGMA user_version` to 5.

3. Implement v6 migration: query branches where `length(context_summary_json) > 51200`. For each, parse JSON, apply `truncate_mid()` to all exchange text fields (user and assistant in first_exchanges and last_exchanges), re-serialize, UPDATE. Batch with per-batch commits. Bump `PRAGMA user_version` to 6.

4. Remove the unconditional `_migrate_project_paths()` call from `get_db_connection()` (now gated inside v5).

5. Remove the bare `except: pass` wrapper around the old `_migrate_project_paths()` call in `get_db_connection()`.

## Test Strategy

- Existing migration tests in `test_db.py` must pass
- New tests in `tests/test_db.py`:
  - `test_v5_migration_enriches_aggregated_content` — create a branch with files_modified JSON, run v5, verify aggregated_content contains file paths
  - `test_v5_migration_renames_interrupted_to_abandoned` — create a branch with "INTERRUPTED" in context_summary, run v5, verify it says "ABANDONED"
  - `test_v5_migration_idempotent` — run v5 twice, verify aggregated_content is identical both times (SET not APPEND)
  - `test_v5_migration_gates_project_paths` — verify `_migrate_project_paths` runs as part of v5, not on every connection
  - `test_v5_fts_rebuild` — verify branches_fts contains updated content after v5
  - `test_v6_truncates_large_json` — create a branch with >50KB context_summary_json, run v6, verify size < 50KB
  - `test_v6_leaves_small_json_unchanged` — branch with <50KB JSON is untouched
  - `test_v6_idempotent` — run v6 twice, verify same result
  - `test_migrations_run_on_get_db_connection` — verify that opening a connection on a v4 database triggers v5 and v6 automatically

## Review Guidance

- **Critical**: v5 must use SET semantics for aggregated_content — verify it recomputes rather than appends
- **Critical**: v5 FTS rebuild must run AFTER all batch UPDATEs are committed, not inside a batch
- Verify batch commit pattern (50 per batch) matches existing backfill_summaries.py pattern
- Verify _migrate_project_paths is removed from the unconditional path and only runs inside v5
- Verify v5 handles NULL files_modified and NULL commits gracefully (many branches have these)
- Check that the INTERRUPTED→ABANDONED rename handles both context_summary (markdown) and context_summary_json (JSON) fields
