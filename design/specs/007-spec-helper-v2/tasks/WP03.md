## Objectives & Success Criteria

`wp-move` uses python-frontmatter for parsing, modifies only the `lane` field (preserving all other metadata including unknown fields), writes atomically via temp file + `os.replace`, and appends activity log entries at the correct section boundary (before the next `##` heading, not EOF). Validation errors in existing metadata produce stderr warnings but do not block the move.

## Subtasks

1. Rewrite `cmd_wp_move` to use `frontmatter.load()` instead of `read_text()` + `parse_frontmatter()`
2. Implement the write path from design.md: `post.metadata["lane"] = new_lane` (direct mutation, no `model_dump()`)
3. Implement atomic write: `NamedTemporaryFile` in same directory + `os.replace`
4. Implement activity log insertion algorithm from design.md:
   - Find `^## Activity Log` via regex (first match if multiple)
   - Find next `^## ` heading after it
   - Insert entry before that heading (or at EOF if none)
   - If no `## Activity Log` section, append `\n\n## Activity Log\n\n` + entry at EOF
5. Entry format: `- {ISO-8601-UTC} — system — lane={new_lane} — moved from {old_lane}`
6. Add stderr warnings: call `validate_wp_metadata` on read, print any errors to stderr, proceed with the move regardless
7. Add stderr warning when current lane equals target lane (no-op)
