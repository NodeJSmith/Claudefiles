## Objectives & Success Criteria

`wp-validate [feature]` validates all WP files against the canonical schema, reports errors/warnings, checks `depends_on` references exist as files, checks `plan_section` references match design.md headers, and supports `--fix` to persist normalization. `wp-list <feature>` outputs frontmatter as JSON for programmatic consumers. Both commands are registered in argparse and dispatched.

## Subtasks

1. Add `wp-validate` subcommand to argparse with optional `feature` positional arg and `--fix` flag
2. Implement `cmd_wp_validate`:
   - For each WP file: load with frontmatter, normalize, validate
   - Report missing required fields, invalid values, broken `depends_on` references
   - Check `plan_section` against actual `## ` headings in `design.md` (if design.md exists in feature dir)
   - Report unknown fields as warnings
   - Report old-schema fields needing normalization as informational
   - With `--fix`: rewrite files with normalized metadata (preserve unknown fields via `post.metadata.update()`)
   - Human output: `"5 files validated, 0 errors, 2 warnings"` + detail lines
   - JSON output: `{"valid": bool, "files": int, "errors": [...], "warnings": [...]}`
3. Add `wp-list` subcommand to argparse with required `feature` positional arg
4. Implement `cmd_wp_list`:
   - Load all WP files, normalize metadata
   - Output JSON array: `[{"wp_id": "...", "title": "...", "lane": "...", "depends_on": [...], "path": "..."}]`
   - Always JSON (no `--human` mode)
5. Wire both commands into the dispatch dict in `main()`
