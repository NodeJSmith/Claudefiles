## Warnings (non-blocking)

1. **AC#6 is claimed by two tasks (T03 and T06) with different meanings.** AC#6's actual text is a behavioral criterion ("Running `/clear` writes a handoff file that the next session's `/cass-resume` can read"). T03 covers the real behavior (the write side, verified against the read contract). T06 also lists AC#6 in `implements`, but its Verify item is about documentation quality, not the underlying behavior — T06's own Summary concedes this is a "partial" claim. This isn't a coverage gap (T03 fully covers the behavioral criterion), but the dual claim slightly muddies the traceability matrix. Consider either dropping `AC#6` from T06's `implements` list (since T06 doesn't implement the behavior) or rephrasing T06's Verify item to something clearly documentation-scoped that doesn't reuse the AC identifier.

2. **T02 and T03 both modify `settings.json` with no dependency edge between them.** T02 removes `extraKnownMarketplaces`/`enabledPlugins`; T03 adds the SessionStart/SessionEnd hook entries. Both depend only on T01, not on each other, so nothing in the task graph forces one to run before the other or serializes the edit. If these run in parallel (e.g., via `mine-orchestrate`), there's a real risk of a merge conflict or a lost edit on the shared file. This is a task-sequencing/decomposition concern, not a requirements-traceability gap — flagging per `decomposition-discipline.md`'s "shared mutable state" check. Recommend adding `T03: depends_on: ["T01", "T02"]` (or vice versa) to force serialization on `settings.json`, or explicitly noting in both tasks' Focus sections that they touch the same file and must not run concurrently.
