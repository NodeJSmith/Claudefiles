# WP03: Create on-demand skills for CLI tool docs

## Objectives & Success Criteria

- [ ] CLI tool documentation moved to on-demand skill(s)
- [ ] Skills load only when Claude needs CLI tool details
- [ ] CLI tool routing tests still pass (no regression)

## Subtasks

1. Decide skill structure — options:
   - **Single skill** `mine.cli-reference` with all CLI tools (simpler, one read)
   - **Grouped skills** by platform: `mine.gh-tools`, `mine.ado-tools`, `mine.session-tools` (more granular loading)
   - Recommend: grouped by platform, since you rarely need GitHub + ADO + tmux docs in the same conversation
2. Create skill directories and SKILL.md files with the CLI documentation extracted from capabilities.md
3. Set `user-invocable: false` on these skills — Claude auto-loads when relevant, zero cost otherwise
4. Verify the routing table one-liners in capabilities.md reference the right skill names
5. Run CLI tool routing tests to confirm no regression

## Test Strategy

Run `evals/compliance/routing/intent-to-cli-tool.yaml`. The routing should still work because the compressed capabilities.md retains one-line entries for each tool, and the full docs are now in discoverable skills.

## Review Guidance

- Skills should contain the full CLI documentation (usage, flags, examples) previously in capabilities.md
- Frontmatter should have `user-invocable: false` and a clear description for Claude's skill discovery
- Avoid bloating skills with content Claude doesn't need — keep it to what's necessary for correct tool invocation
