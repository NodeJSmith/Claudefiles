## Objectives & Success Criteria

Rewrite `findings-protocol.md` and `caller-protocol.md` to define the streamlined findings contract. These two files are the foundation — WP02 (SKILL.md) and WP03 (caller updates) both reference them. Success: both files are internally consistent, the findings file format includes `status` and `overflow` fields, all manifest/editor/verb documentation is removed, and `findings-protocol.md` is ≤150 lines and `caller-protocol.md` is ≤100 lines.

## Subtasks

1. Read the current `skills/mine.challenge/findings-protocol.md` (418 lines) to understand the existing format contract — identify which sections define the findings file format vs. the manifest/editor flow
2. Rewrite `skills/mine.challenge/findings-protocol.md` to document only:
   - The findings file format (markdown with `## Finding N:` headings)
   - Required fields per finding: severity, type, design-level, resolution, raised-by, why-it-matters, evidence, references, design-challenge
   - New fields: `status: applied | pending | overflow | skipped`, `overflow: true | false`
   - The auto-apply classification criteria (when resolution=auto-apply: localized fix, critic agreement, not CRITICAL)
   - The inline resolution flow: auto-apply findings executed silently, user-directed findings presented one at a time via AskUserQuestion
   - The finding cap: CRITICAL/HIGH always shown, MEDIUM budgeted, overflow handling
   - Remove entirely: resolution manifest (resolutions.md), verb vocabulary (fix/A/B/C/skip/defer/ask/file), consent gate, editor session, hash-based change detection, compaction recovery via manifest
3. Read the current `skills/mine.challenge/caller-protocol.md` (363 lines) to understand the existing caller contract
4. Rewrite `skills/mine.challenge/caller-protocol.md` to document only:
   - How callers invoke challenge (`--findings-out`, `--target-type`, `--mode`)
   - What callers receive: the findings file with status fields populated
   - How callers should handle `status: applied` findings (verify the edit was applied correctly)
   - How callers should handle `status: pending` findings (already resolved during challenge execution)
   - How callers should handle `status: overflow` findings (auto-applied or deferred)
   - The `design-level` field and its meaning for doc-edit callers
   - Remove entirely: pre-routing tables, default verb selection, manifest generation, editor session, Consent Gate, Detection/Validation/Commit Gate, post-execute hooks, compaction recovery via manifest hash
