## Objectives & Success Criteria

After Phase 2 discovery closes (confirm intent) and before Phase 3 research dispatch, mine.define revisits its Phase 1.5 codebase findings and presents a structured sub-problem → existing code table. The table decomposes the user's confirmed intent into 3-7 sub-problems and maps each to existing code with Full/Partial/None coverage. The Architecture section in Phase 4 must reference existing code from Full and Partial rows. Phase 1.5 itself remains unchanged (free-form summary).

## Subtasks

1. In `skills/mine.define/SKILL.md`, after the "Confirm intent summary" block (after line 186) and before Phase 3 (line 190), insert a new subsection `### Existing code leverage (moderate+ only)`:
   - Instruction to decompose the user's confirmed intent into 3-7 sub-problems using discovery answers, scope mode, and Phase 1.5 code findings
   - Markdown table format: `| Sub-problem | Existing code | Coverage |` with three coverage levels: "Full — reuse as-is", "Partial — handles X but not Y", "None — new code needed"
   - Present the table with: "Here's what I found. Sub-problems with existing coverage should reuse that code rather than rebuilding. Correct me if I'm wrong about any of these."
   - Skip for trivial features (Phase 1.5 doesn't run for trivial, so no code findings to revisit)
   - Edge case: if Phase 1.5 found no existing code, present an empty table with a note

2. In the Phase 4 "Write design.md" section (around line 260), add a rule to the Architecture section guidance: "Reference existing code from the sub-problem table's Full and Partial rows. If the architecture proposes new code for a sub-problem marked Full, justify why."

N/A — no test infrastructure in this repo. Validation is manual: invoke `/mine.define` with a moderate feature in a repo with existing code, verify the sub-problem table appears after confirm intent and before research dispatch.

- Verify Phase 1.5 is NOT modified — it keeps its free-form summary. The structured table is a separate step that revisits those findings
- Verify the Architecture section rule is added: must reference Full/Partial rows, must justify deviations from Full coverage
- Verify trivial features skip the table (consistent with Phase 1.5 skip)
- Verify the empty-table edge case is handled (no existing code found)
