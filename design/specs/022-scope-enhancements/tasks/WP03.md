## Objectives & Success Criteria

After Phase 2 discovery closes (confirm intent) and before Phase 3 research dispatch, mine.define revisits its Phase 1.5 codebase findings and presents a structured sub-problem → existing code table. The table decomposes the user's confirmed intent into 3-7 sub-problems and maps each to existing code with Full/Partial/None coverage. The Architecture section in Phase 4 must reference existing code from Full and Partial rows. Phase 1.5 itself remains unchanged (free-form summary).

## Subtasks

1. In `skills/mine.define/SKILL.md`, after the `### Confirm intent summary` AskUserQuestion block and before `## Phase 3: Investigate`, insert a new subsection `### Existing code leverage (moderate+ only)`:

   - Instruction to decompose the user's confirmed intent into 3-7 sub-problems using discovery answers, scope mode, and Phase 1.5 code findings
   - Present a structured markdown table:

     ```markdown
     | Sub-problem | Existing code | Coverage |
     |---|---|---|
     | Validate user email | `src/validators.py` — has `validate_email()` | Full — reuse as-is |
     | Rate limit API calls | `src/middleware.py` — rate limiter exists but only for auth endpoints | Partial — handles auth but not general API |
     | Send notification on failure | (none found) | None — new code needed |
     ```

   - Coverage vocabulary: `Full — reuse as-is` (existing code solves this entirely), `Partial — <what's missing>` (existing code solves part of it), `None — new code needed` (nothing found)
   - Present with: "Here's what I found. Sub-problems with existing coverage should reuse that code rather than rebuilding. Correct me if I'm wrong about any of these."
   - Skip for trivial features (Phase 1.5 doesn't run for trivial, so no code findings to revisit)
   - Edge case: if Phase 1.5 found no existing code, present an empty table with a note ("no existing code found for any sub-problem")

2. In the `### Write design.md` section, add a rule to the Architecture section guidance: "Reference existing code from the sub-problem table's Full and Partial rows. If the architecture proposes new code for a sub-problem marked Full, justify why."

## Test Strategy

N/A — no test infrastructure in this repo. Validation is manual: invoke `/mine.define` with a moderate feature in a repo with existing code, verify the sub-problem table appears after confirm intent and before research dispatch.

- Verify the empty-table edge case is handled (no existing code found)
- FAIL if: Phase 1.5 output format is changed, or if the sub-problem decomposition runs before Phase 2 closes
