## Objectives & Success Criteria

After the user answers problem grounding (Q1) and success definition (Q2), mine.define presents a scope mode selection (expand/hold/reduce). The selected mode shapes all subsequent Phase 2 questions (Q3 onward) and Phase 4 design doc writing. The mode is persisted in the design doc header as `**Scope-mode:**` and is visible in question headers throughout the session. Trivial features skip scope mode selection entirely.

## Subtasks

1. In `skills/mine.define/SKILL.md` Phase 2, after the success definition AskUserQuestion block and before the `### Ask for moderate and complex features` heading, insert a new subsection `### Scope mode selection (moderate+ only)` with this AskUserQuestion block:

   ```
   AskUserQuestion:
     question: "You described the problem as [X] and success as [Y]. Given that and what I found in the codebase, how should we scope this?"
     header: "Scope mode"
     multiSelect: false
     options:
       - label: "Expand — build the ambitious version"
         description: "Push scope up. What would make this 10x better? What adjacent improvements would make it sing?"
       - label: "Hold — make this bulletproof"
         description: "Accept the scope as stated. Focus on making it solid, complete, and well-tested."
       - label: "Reduce — strip to essentials"
         description: "Find the minimum viable version. Cut everything that isn't core. What can be a follow-up?"
   ```

   Where `[X]` and `[Y]` are the user's actual answers to Q1 (problem grounding) and Q2 (success definition).

   - Skip for trivial features
   - On resume from existing feature directory: check design.md header for `**Scope-mode:**` field; if present, skip re-asking and announce the recovered mode

2. Add mode effects on Q3+ questions. Modify the existing question blocks after scope mode selection to include mode-specific variants. The mode-specific rewordings:

   **Scope boundary** (the `### Ask for moderate and complex features` section, first question):
   - Expand: "What's phase 1 vs phase 2? What should we build now, and what's a natural follow-on?"
   - Hold: unchanged — "Anything I should explicitly NOT include?"
   - Reduce: "What can we cut entirely? What's the absolute minimum that ships value?"

   **User flow** (the `### Ask for moderate and complex features` section, second question):
   - Expand: Ask the standard question, then follow up: "Are there adjacent flows or related scenarios we should include?"
   - Hold: unchanged — "Walk me through the main scenario"
   - Reduce: Ask the standard question, then follow up: "Which of these steps could be manual or deferred for now?"

   **Adaptive follow-ups** (the `### Adaptive follow-up` section):
   - Expand: more follow-ups than the tier suggests; explore opportunity branches
   - Hold: standard follow-ups per the complexity tier
   - Reduce: fewer follow-ups; bias toward deferring unresolved branches rather than probing deeper

3. Add anti-drift rule after the scope mode selection block:
   - Phase 2 question headers prefixed with mode: `[Expand] Non-goals`, `[Hold] User flow`, `[Reduce] Edge cases`
   - The `### Confirm intent summary` section must include `**Scope mode:** <mode>` in the summary so the user can detect drift before the design doc is written
   - Note: "If a later question or finding suggests a different mode would be better, note it once — do not act on it unless the user explicitly changes mode."

4. Add mode effects on Phase 4 design doc writing. In the `### Write design.md` section, add mode-specific guidance for these sections:
   - Problem: Expand → include the broader problem, not just the immediate one; Hold → state as given; Reduce → state the acute problem only
   - Goals: Expand → include stretch goals alongside core goals; Hold → core goals only; Reduce → minimum viable goals only
   - Architecture: Expand → include platform opportunities, extensibility points; Hold → standard recommendation; Reduce → simplest possible approach, documents only what IS being built
   - Non-goals: Expand → frame as "what's phase 2 vs phase 1?"; Hold → as-is; Reduce → explicitly list cut items with rationale (mine.plan uses Non-goals as exclusions)
   - Alternatives: Expand → include the ambitious alternative even if not chosen; Hold → standard alternatives; Reduce → include "do nothing" and "manual workaround" as alternatives

5. Add `**Scope-mode:** <expand|hold|reduce>` to the design doc template header block (after `**Status:** draft`). This field persists the mode for resume detection and downstream visibility.
