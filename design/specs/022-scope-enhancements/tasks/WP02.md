## Objectives & Success Criteria

After the user answers problem grounding (Q1) and success definition (Q2), mine.define presents a scope mode selection (expand/hold/reduce). The selected mode shapes all subsequent Phase 2 questions (Q3 onward) and Phase 4 design doc writing. The mode is persisted in the design doc header as `**Scope-mode:**` and is visible in question headers throughout the session. Trivial features skip scope mode selection entirely.

## Subtasks

1. In `skills/mine.define/SKILL.md` Phase 2, after the success definition question block (after line 93) and before the "Ask for moderate and complex features" section (line 95), insert a new subsection `### Scope mode selection (moderate+ only)`:
   - AskUserQuestion block with three options (Expand, Hold, Reduce) per the design doc. The question text must reference the user's problem and success answers: "You described the problem as [X] and success as [Y]. Given that and what I found in the codebase, how should we scope this?"
   - Skip for trivial features
   - On resume from existing feature directory: check design.md header for `**Scope-mode:**` field; if present, skip re-asking and announce the recovered mode

2. Add mode effects on Q3+ questions. Modify the existing question blocks for scope boundary (line 97), user flow (line 105), and adaptive follow-ups (line 153) to include mode-specific variants:
   - Scope boundary: Expand → "what's phase 2 vs phase 1?"; Hold → as-is; Reduce → "what can we cut entirely?"
   - User flow: Expand → probe for adjacent flows; Hold → as-is; Reduce → "which steps could be manual for now?"
   - Adaptive follow-ups: Expand → more follow-ups, explore opportunities; Hold → standard; Reduce → fewer follow-ups, bias toward deferring branches

3. Add anti-drift rule after the scope mode selection block:
   - Phase 2 question headers prefixed with mode: `[Expand] Non-goals`, `[Hold] User flow`, `[Reduce] Edge cases`
   - Confirm intent summary (line 169) must include `**Scope mode:** <mode>`
   - Note: "If a later question or finding suggests a different mode would be better, note it once — do not act on it unless the user explicitly changes mode."

4. Add mode effects on Phase 4 design doc writing. In the "Write design.md" section (line 260), add mode-specific guidance for these sections:
   - Problem: Expand → broader problem; Hold → as given; Reduce → acute problem only
   - Goals: Expand → include stretch goals; Hold → core only; Reduce → minimum viable only
   - Architecture: Expand → platform opportunities, extensibility; Hold → standard; Reduce → simplest approach, documents only what IS being built
   - Non-goals: Expand → frame as phase 2 vs phase 1; Hold → as-is; Reduce → explicitly list cut items with rationale
   - Alternatives: Expand → include ambitious alternative; Hold → standard; Reduce → include "do nothing" and "manual workaround"

5. Add `**Scope-mode:** <expand|hold|reduce>` to the design doc template header block (line 264, after `**Status:** draft`). This field persists the mode for resume detection and downstream visibility.

- Verify trivial features skip scope mode entirely
- Verify the AskUserQuestion references the user's actual problem/success answers, not just recon findings
- Verify mode-specific question variants only apply to Q3+ (scope boundary onward), never Q1 or Q2
- Verify anti-drift: mode appears in question headers, confirm intent summary, and design doc header
- Verify resume path: design doc header `**Scope-mode:**` is checked on resume and skips re-asking
- FAIL if: scope mode fires before Q1/Q2, or if Q1/Q2 have mode-specific variants, or if the design doc template lacks the Scope-mode header field
