## Objectives & Success Criteria

The orchestrator writes a checkpoint file at `<feature_dir>/tasks/.orchestrate-state.md` after Phase 0 completes and after each WP verdict. On startup, it detects an existing checkpoint and offers resume/restart. The checkpoint uses a hybrid write strategy: full rewrite for the header key-value section, append-only for verdict blocks.

Success:
- Checkpoint file is created with correct key-value fields after Phase 0
- Checkpoint file is updated with an appended verdict block after each WP verdict (header rewritten, verdict appended)
- Resume flow detects the checkpoint and offers the user resume/restart options
- On resume, orchestrator restores state and skips to the next WP after `last_completed_wp`
- On restart, checkpoint file is deleted
- Staleness detection: checkpoint older than 24 hours defaults to "restart" in the prompt
- Checkpoint is deleted after the post-orchestration review results gate

## Subtasks

1. Add a new section at the top of Phase 0 in `skills/mine.orchestrate/SKILL.md` (before "Find the feature directory"): check for `<feature_dir>/tasks/.orchestrate-state.md`. If found, read it and present the resume/restart AskUserQuestion. Handle both paths (resume: restore state + skip ahead; restart: delete file + proceed normally).
2. Add checkpoint initial write after Phase 0 completes (after dev server check): Write the checkpoint file with all key-value fields (feature_dir, tmpdir, visual_skip, dev_server_url, base_commit from `git rev-parse --short HEAD`, started_at as ISO timestamp). Verdicts section starts empty.
3. Add checkpoint update after each WP verdict at Step 9: rewrite the header section (update last_completed_wp, warn_counter), then append a new verdict block with the WP's ID, title, verdict, commit SHA, and optional notes.
4. Add checkpoint deletion at the end of Phase 3 (after the user chooses at the review results gate — the very last step).
5. On resume: reset `warn_counter` to 0, verify tmpdir exists (create new if missing), re-read design doc and all WP files.

## Test Strategy

This is a SKILL.md prompt change — no automated tests. Verification is manual: run `/mine.orchestrate` on a 2-WP test feature, verify the checkpoint file is created and updated correctly, then simulate a resume by restarting the session and running `/mine.orchestrate` again on the same feature.

## Review Guidance

- **Spec reviewer**: Verify the checkpoint format matches the design doc exactly (append-only verdicts, hybrid write strategy). Confirm resume flow handles missing tmpdir gracefully.
- **Code reviewer**: Check that the checkpoint write instructions are unambiguous — the LLM must know exactly when to rewrite the header vs. append a verdict block. Verify no race between `spec-helper wp-move` and checkpoint write.
- **Integration reviewer**: Ensure the resume flow doesn't conflict with the existing Phase 1 auto-select behavior (`SKILL.md:98` — "if any WP is in `lane: doing`, resume that WP first"). The checkpoint's `last_completed_wp` and the frontmatter's lane state must agree.
