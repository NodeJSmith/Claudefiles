## Objectives & Success Criteria

Phase 3 is rewritten to run implementation-review and challenge automatically with minimal user prompts. The user is only prompted when blocking issues are found (impl-review) or at the final challenge results gate.

Success:
- Phase 3 summary prints automatically (kanban + verdict table from checkpoint)
- Implementation-review is invoked automatically (no "Run it?" prompt)
- If impl-review returns APPROVE → auto-challenge runs automatically
- If impl-review returns REQUEST_FIXES/ABANDON → user is prompted immediately to address issues before challenge
- Auto-challenge is dispatched as a subagent (for context isolation) using full Opus model, scoped to `git diff --name-only <base_commit> HEAD`
- Challenge findings + impl-review suggestions presented together in a single prompt
- "Address findings" dispatches a fresh executor subagent (not inline orchestrator fixes), then re-runs code-reviewer + integration-reviewer on the fix diff, then re-runs the challenge
- "Accept and ship" invokes `/mine.ship`
- Checkpoint is deleted after the user chooses at this final gate
- The `mine.implementation-review/SKILL.md` prompt for "Run implementation-review?" is removed (orchestrator invokes it directly)

## Subtasks

1. Rewrite Phase 3 in `skills/mine.orchestrate/SKILL.md` (`SKILL.md:412-444`). Replace the current summary + "Run implementation-review?" gate with the three-step pipeline: Step 1 (summary), Step 2 (auto impl-review with blocking-issues gate), Step 3 (auto-challenge with final prompt).
2. For Step 2, add the impl-review blocking-issues gate: if REQUEST_FIXES or ABANDON, prompt the user. On "Address fixes", dispatch a fresh executor subagent with the findings, re-run reviewers, re-run impl-review. If APPROVE, continue to Step 3.
3. For Step 3, add the auto-challenge dispatch. The challenge must be dispatched as a single `general-purpose` subagent that runs `/mine.challenge` internally (nested subagents for the three critics). The orchestrator passes: `base_commit`, changed file list from `git diff --name-only <base_commit> HEAD`, design doc path, and output findings file path. The orchestrator reads the findings file after the subagent completes.
4. For the final challenge results prompt, present combined findings (challenge + any impl-review non-blocking suggestions). Handle "Address findings" (fresh executor subagent → re-review → re-challenge loop), "Accept and ship" (`/mine.ship`), and "Stop here" (checkpoint persists).
5. Add checkpoint deletion after the user chooses at this gate.
6. Update `skills/mine.implementation-review/SKILL.md` Phase 4 gate to remove the standalone "Run implementation-review?" prompt when invoked by the orchestrator. The implementation-review skill should detect that it was invoked programmatically (by the orchestrator passing it as a subagent) vs. standalone, and skip its own gate accordingly.

## Test Strategy

Manual verification: run `/mine.orchestrate` on a 2-WP test feature and verify the full post-orchestration pipeline runs: summary → auto impl-review → auto challenge → single user prompt with combined results.

## Review Guidance

- **Spec reviewer**: Verify the three-step pipeline matches the design doc exactly. Check that the impl-review blocking gate happens before the challenge (not after). Verify the challenge is dispatched as a subagent with correct isolation.
- **Code reviewer**: Check that the fresh executor subagent for fixes gets clean context (only findings + file paths, not the full orchestration history). Verify the re-review loop after fixes includes both code-reviewer and integration-reviewer.
- **Integration reviewer**: Confirm the changes to `mine.implementation-review/SKILL.md` don't break standalone invocations of `/mine.implementation-review`. The skill must work both when invoked by the orchestrator and when invoked directly by the user.
