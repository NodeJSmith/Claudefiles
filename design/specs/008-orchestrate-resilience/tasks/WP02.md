## Objectives & Success Criteria

After each WP passes all review gates, the orchestrator creates a WIP commit before updating the checkpoint. The `base_commit` is recorded at orchestration start. Each verdict block in the checkpoint includes the WIP commit SHA.

Success:
- `base_commit` is captured via `git rev-parse --short HEAD` at orchestration start and written to the checkpoint
- After each WP passes Step 9, `git add -A && git commit -m "WIP: WP<NN> -- <title>"` is executed
- The WIP commit SHA is captured and written to the verdict block's `commit` field
- `git diff --name-only <base_commit> HEAD` returns meaningful results after at least one WP completes

## Subtasks

1. In Phase 0 of `skills/mine.orchestrate/SKILL.md`, after the initial checkpoint write (from WP01), record `base_commit` via `git rev-parse --short HEAD`. This is already part of the checkpoint write — ensure the instruction is explicit about timing (before any WP execution begins).
2. In Step 10 of the per-WP execution loop, after `spec-helper wp-move` moves the WP to `done`, add a WIP commit step: `git add -A && git commit -m "WIP: WP<NN> -- <title>"`. Then capture the new HEAD SHA via `git rev-parse --short HEAD` and include it in the checkpoint verdict block write.
3. Ensure the WIP commit happens *before* the checkpoint update — the checkpoint's verdict block must include the commit SHA from the WIP commit, not from before it.

## Test Strategy

Manual verification: run `/mine.orchestrate` on a 2-WP test feature and verify `git log --oneline` shows two WIP commits with correct messages. Verify `git diff --name-only <base_commit> HEAD` returns the changed files.

## Review Guidance

- **Spec reviewer**: Verify the ordering: review gates pass → `spec-helper wp-move` → WIP commit → checkpoint update. All four steps must happen in this order.
- **Code reviewer**: Check that the `git add -A` doesn't accidentally stage unrelated files (e.g., the checkpoint file itself). The checkpoint file should be gitignored per the design.
- **Integration reviewer**: Confirm the WIP commit doesn't interfere with the code-reviewer loop in Step 7, which uses `git diff --name-only` to find changed files. After a WIP commit, the next WP's code reviewer should only see that WP's changes, not the previous WP's.
