# Prior Art: Humanizing AI-Generated Prose

## The Problem

AI-generated text has structural and vocabulary fingerprints that erode reader trust even when the content is correct. PR descriptions, commit messages, research briefs, and documentation accumulate these tells: "delve," em-dash chains, tricolon lists, negative parallelisms ("it's not X -- it's Y"), resolution closers that restate what was just said. Readers develop an instinct for it and discount the text, even subconsciously.

Prevention rules (loaded into context at generation time) help but don't catch everything. Text generated by subagents, workflows, or earlier sessions still needs post-processing. The gap is an active editing skill that can clean up prose after the fact.

## How We Do It Today

We have `writing-quality.md` as a passive rule loaded into every session. It names 22+ patterns with mechanisms, includes an "Adding Voice" section, and ends with a self-audit prompt. It's comprehensive for prevention but has no active enforcement -- no skill you invoke to rewrite or audit text that's already been generated. `mine.clean-code` handles code quality but explicitly excludes prose.

## Patterns Found

### Pattern 1: Banned-Word Lists with Tiered Alternatives

**Used by**: stop-slop (5,000+ stars), avoid-ai-writing, humanize-writing-skill, tropes.fyi, our writing-quality.md
**How it works**: Curated lists of AI-tell words mapped to plain alternatives, organized by severity tier. "Delve" is always wrong in a commit message; "robust" is fine in engineering docs describing actual robustness. The tiering prevents over-correction.
**Strengths**: Easy to implement, validate, and extend. Catches the most obvious tells.
**Weaknesses**: Misses structural tells entirely. Lists drift as models change -- today's tells become tomorrow's fine-tuned-away patterns while new ones emerge.
**Example**: https://github.com/conorbronsdon/avoid-ai-writing (109-entry, 3-tier table)

### Pattern 2: Multi-Pass Editing Architecture

**Used by**: theclaymethod/unslop (2-pass), avoid-ai-writing (2-pass), humanize-writing-skill (3-pass)
**How it works**: Decompose editing into sequential passes. Common split: Pass 1 vocabulary, Pass 2 structure, Pass 3 texture/voice. Fixing vocabulary in pass 1 exposes structural problems that were previously masked, and structural fixes in pass 2 can create new vocabulary tells. The second/third pass catches these cascading effects.
**Strengths**: Catches patterns single-pass approaches miss. Each pass has a focused concern.
**Weaknesses**: More expensive (multiple LLM calls). Risk of over-editing past 2-3 passes. Diminishing returns.
**Example**: https://github.com/lguz/humanize-writing-skill (3-pass with distinct concerns)

### Pattern 3: Structural Pattern Detection (Beyond Words)

**Used by**: tropes.fyi, Bloomberry research, Wikipedia editors, stop-slop
**How it works**: Detects sentence-level and paragraph-level structural fingerprints: negative parallelism, AI Sentence DNA (4-part arc: claim, expansion, contrast, resolution), tricolon abuse, bolded-header bullets restating the bold text, copula avoidance, resolution closers. Bloomberry found 82% of AI-generated posts share these 4 structural fingerprints regardless of model.
**Strengths**: Catches tells that survive vocabulary cleanup. More stable across model versions than word choices.
**Weaknesses**: Harder to implement as automated rules. Higher false-positive rate since some structures appear in human writing too.
**Example**: https://tropes.fyi/tropes-md (named patterns with mechanisms)

### Pattern 4: Preserve-Then-Edit (Technical Content Protection)

**Used by**: theclaymethod/unslop, avoid-ai-writing
**How it works**: Before editing, lock regions that must not be modified: code blocks, inline code, URLs, headings, YAML frontmatter, tables, blockquotes. Only prose between protected regions is eligible for rewriting. Extended by some to "already-human passages" -- text that doesn't trigger any flags is left untouched.
**Strengths**: Critical for PR descriptions and docs where technical accuracy in code references and commands is non-negotiable.
**Weaknesses**: Requires robust region detection. Edge cases around code comments, headings with AI-tell words.
**Example**: https://github.com/theclaymethod/unslop (byte-identical preservation)

### Pattern 5: Operational Modes (Rewrite vs. Detect vs. Edit)

**Used by**: avoid-ai-writing (3 modes), mshumer/unslop (profile + edit)
**How it works**: Three distinct modes: **Detect** flags patterns without changing text (dry-run / audit). **Edit** makes minimal, targeted in-place changes, preserving passages that are already human. **Rewrite** restructures and replaces wholesale for drafts where you want maximum de-slopping.
**Strengths**: Matches the tool to the situation. A commit message needs surgical edits; a research brief draft might benefit from full rewrite.
**Weaknesses**: More complex skill with more entry points. Users need to know which mode to choose.
**Example**: https://github.com/conorbronsdon/avoid-ai-writing

### Pattern 6: Prevention vs. Post-Processing

**Used by**: tropes.fyi (system prompt insert), our writing-quality.md
**How it works**: Suppress AI patterns at generation time by loading anti-slop rules into context. More efficient than generate-then-fix because the model has access to full meaning during generation. Our writing-quality.md is already this approach.
**Strengths**: No additional LLM calls. Avoids information loss from post-processing.
**Weaknesses**: Doesn't catch everything. Can't fix text from other tools or subagents. Requires context window space.
**Example**: https://tropes.fyi/tropes-md

## Anti-Patterns

- **Detection-bypass as goal**: Commercial "humanizer" tools (HumanizerAI, BypassGPT) optimize for fooling AI detectors rather than improving prose. Some use Unicode substitution or synonym cycling that makes text worse. The goal should be prose that reads well to humans, not prose that fools classifiers.

- **Over-correction into sterile voice**: Skills that only subtract (remove words, cut structures) without adding anything back produce flat, lifeless prose -- a different kind of obvious. "Removing patterns is half the job."

- **Static lists without mechanism understanding**: Flat banned-word lists without explaining why each is a tell lead to cargo-cult editing. "Delve" isn't inherently wrong -- it's a tell because LLMs use it 400% more than humans in the same contexts.

- **Treating all text types equally**: A commit message, PR description, research brief, and blog post have different registers. No tool found explicitly adapts rules to text type -- this is an open gap.

## Emerging Trends

The field is converging on **named-pattern taxonomies** with shared vocabulary (negative parallelism, tricolon abuse, copula avoidance). **Structural detection** is emerging as more durable than vocabulary detection -- 82% cross-model consistency for structural fingerprints vs. vocabulary tells that get fine-tuned away. And **agent skills are the primary distribution channel** -- every major tool ships as a skill file, not a standalone app.

## Relevance to Us

Our `writing-quality.md` is already one of the better prevention-side implementations -- it names patterns with mechanisms, includes "Adding Voice" guidance, and covers patterns (colon overuse, inline-header lists) that don't appear in other tools. What we lack is the active editing side:

1. **No invocable skill** -- writing-quality.md is passive context, not something you run on text
2. **No multi-pass architecture** -- the self-audit prompt is a single mental pass
3. **No detect/audit mode** -- no way to scan text for AI tells without rewriting it
4. **No technical content protection** -- no explicit preserve-code-blocks logic for editing around technical content
5. **No text-type awareness** -- same rules regardless of whether you're editing a commit message or a research brief

The strongest design would combine our existing pattern taxonomy (which is already good) with avoid-ai-writing's three-mode architecture, a two-pass editing approach, and explicit content-type awareness for our target formats (PR descriptions, commit messages, docs, research briefs).

## Recommendation

Build a new skill (working name: something like `mine.unslop` or `mine.prose-edit`) that:

1. **Leverages writing-quality.md** as its pattern reference rather than building a new taxonomy from scratch
2. **Offers three modes**: detect (audit without changes), edit (minimal surgical fixes), rewrite (full de-slop + voice injection)
3. **Defaults to edit mode** since our primary use case is polishing mostly-good text
4. **Protects technical content** (code blocks, YAML, URLs, tables) byte-identical through edits
5. **Uses two-pass editing**: pass 1 fixes vocabulary + structure, pass 2 re-reads for cascading tells and adds voice
6. **Is text-type aware**: different registers for commit messages (terse, imperative) vs. PR descriptions (narrative) vs. research briefs (analytical)

The pstack unslop skill and our writing-quality.md cover the same ground conceptually -- the value-add is making it an active, invocable editing tool with mode selection and content-type awareness, which nothing in the ecosystem does well yet.
