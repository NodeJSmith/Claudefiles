2 — Context & Compaction
PreCompact Hook Pattern
pilot-shell · claude-context-mode · arscontexta
Three independent sources converge on the same idea: automate context preservation before compaction fires, don't rely on users to run it manually. Pilot-shell uses a pre_compact.py hook that captures active plans, task state, and decisions to persistent memory. Claude-context-mode builds priority-tiered XML snapshots (≤2KB) and indexes session events into SQLite FTS5 for on-demand retrieval. Arscontexta uses a Stop hook for session capture. Your /mine.pre-compact requires manual invocation — this is the single biggest gap.
Session Continuity Hooks
claude-context-mode · pilot-shell
Claude-context-mode uses four hooks working together: PostToolUse (captures edits, git ops, errors), UserPromptSubmit (records decisions), PreCompact (builds snapshot), SessionStart (restores state). Events indexed into FTS5 and retrieved via BM25 search — not dumped back into context. This is more sophisticated than your current approach.
Fresh Context per Iteration
ralph loop · ralph for idiots · sysls
Ralph's core insight: "Progress persists. Failures evaporate." Each iteration starts clean; state lives in files + git, not context. Your orchestrate skill already does this (subagents get fresh context), but the explicit framing of "context pollution as certainty, not accident" is useful. STEERING.md for mid-run redirection is a transferable pattern.
3 — Review Quality
10-Section Plan Review
Garry Tan mega-plan (765 lines)
Your plan-review has 6 checkpoints. Garry's adds: Error & Rescue Registry (METHOD | WHAT CAN GO WRONG | EXCEPTION CLASS | RESCUED? | USER SEES table), Failure Modes Registry (unrescued + untested + silent = CRITICAL GAP), observability strategy, deployment safety, long-term trajectory. Three scope modes (EXPANSION / HOLD SCOPE / REDUCTION) change review expectations. The "zero silent failures" principle: every rescue StandardError is a smell.
Scope Challenge & Anti-Drift
Garry Tan plan-exit-review (151 lines)
Three mandatory questions before analysis: 1) What existing code already solves this? 2) What's the minimum viable change? 3) If touching >8 files or introducing >2 new classes, treat as smell. Then the anti-scope-drift instruction: "Once I've chosen scope, commit fully. Do not silently reduce scope, skip components, or re-argue for less work." This addresses a real Claude behavior.
Immutable Decision Logs
Codified Context (arxiv)
Once an agent records a choice, subsequent agents read it before proposing alternatives. Conflict detection: new recommendations contradicting prior entries trigger resolution protocol. Checkpoint validation: at session boundaries, verify inherited context matches actual codebase state. Your design docs freeze after plan-review, but mid-session decisions aren't logged immutably.
4 — TDD Enforcement
One-Test-At-A-Time Vertical Slicing
TDD Skill (aihero.dev) · pilot-shell
Anti-pattern: Write 5 tests, then implement all 5 (horizontal). This allows "dishonest" tests that verify imagined behavior.

Required: RED→GREEN for test1→impl1, then RED→GREEN for test2→impl2. One test at a time. Only enough code to pass current test. Don't anticipate future tests.

The TDD skill is a 6-file directory (SKILL.md + tests.md + deep-modules.md + interface-design.md + mocking.md + refactoring.md). Enforcement is purely instructional — no hook. Pilot-shell adds a PostToolUse hook that checks if implementation files were modified without a failing test first.
Interface Design for Testability
TDD Skill (aihero.dev)
Three rules: 1) Accept dependencies, don't create them (DI). 2) Return results, don't produce side effects. 3) Small surface area (fewer methods = fewer tests). Mock only at system boundaries: external APIs, databases, time/randomness, file system. Never mock your own classes.
5 — Context Bloat Reduction
CLAUDE.md as Routing Table
sysls · best-practice repo · context engineering skills
Multiple sources converge: "CLAUDE.md should be as barebones as possible, only containing IF-ELSE of where to find context." Best-practice repo recommends under 200 lines. Your CLAUDE.md is 114 lines (good), but your rules files total ~1,700 lines loaded into every conversation. The attention degradation research (lost-in-the-middle effect, U-shaped curves) suggests information in the middle of long context gets less attention. Critical instructions should be at top or bottom.
On-Demand vs Always-Loaded
Everything is Context (arxiv) · sysls · Seeing like an Agent
The arxiv paper proposes lazy materialization: don't load context until agent requests it. The Anthropic team confirms: progressive disclosure > stuffing the system prompt. Your frontend-workflow.md (~50 lines) loads for every conversation but only matters for UI work. Candidates for on-demand loading: frontend-workflow, backlog convention, tmux naming, command-output preservation.
Rules/Skills Consolidation
sysls · best-practice repo
"As you add more rules and skills, they start to contradict each other. Tell your agents to go for a spa day and consolidate." Periodic audit to remove contradictions, merge overlapping rules, and compress verbose examples. Your /mine.audit does this for code but not for your own rules/skills.
6 — Novel Patterns
Adversarial Multi-Agent
sysls
Bug-finder (scored +1/+5/+10 by severity) → reports superset of all possible bugs.
Adversarial agent (gets bug-score for disproved bugs, -2x penalty for wrong) → aggressively disproves.
Referee (told "I have ground truth, +1/-1") → scores both. "Frighteningly high fidelity." Exploits sycophancy by design.
Guardrails.md (Append-Only)
ralph for idiots
Same mistake never happens twice. When something breaks, agent appends a "sign" with trigger, instruction, and context. Guardrails are append-only, read at start of each iteration. More disciplined than your error-tracking rule (which is per-session and ephemeral). Gutter detection: automated signal when same failure repeats.
Task Completion Contracts
sysls
Formalize "done" as a contract: tests pass, screenshot verification, specific criteria met. Stop hook prevents session termination until contract is fulfilled. Your WP system has success criteria but no stop-hook enforcement. New session per contract > long-running sessions.
Skill Graphs
arscontexta (Heinrich)
Skills as interconnected graph nodes with wikilinks. Progressive disclosure: index→YAML frontmatter→links→full content. Most decisions happen before reading a full file. Premature at 32 skills but worth revisiting at 50+. Your capabilities.md intent-routing table already implements the index pattern.

9 — Cross-Cutting Patterns

Themes that appeared across 3+ independent sources, suggesting strong consensus.

1
Context pollution is inevitable, not accidental. Ralph, sysls, pilot-shell, and both arxiv papers all frame context degradation as a certainty to engineer around, not a bug to fix. Design for fresh context (subagents, iteration boundaries) rather than trying to maintain a single long-lived session.
2
Progressive disclosure beats upfront loading. The Anthropic team (Seeing like an Agent), context engineering skills, skill graphs, and best-practice repo all validate: load skill names initially, full content on relevance. Your capabilities.md routing table already implements this — extend to rules files.
3
Separate research from implementation. sysls, TDD skill, Garry Tan, and pilot-shell all emphasize: research pollutes implementation context. Use a fresh agent (or subagent) for implementation after research concludes. Your mine.research → mine.design → mine.orchestrate pipeline already does this well.
4
State lives in files and git, not in context. Ralph, pilot-shell, Codified Context paper, and beads all converge: persistent state (plans, decisions, task status) must be externalized to files. Context-resident state is volatile and unreliable. Your WP lane state in YAML frontmatter and design docs already follow this principle.
5
Revisit tool assumptions as models improve. The Anthropic team explicitly: "Tools that models once needed might now be constraining them." sysls: frontier companies will incorporate anything truly useful. Implication: periodically audit whether your rules and agents are still necessary — models may have internalized the behavior.
6
Exploit sycophancy, don't fight it. sysls's adversarial pattern, Garry Tan's anti-scope-drift instruction, and the TDD skill's "one test prevents dishonest testing" all work with model tendencies rather than against them. Design agent interactions that channel the desire to please into productive outcomes.
